# Contributing

name-please is an opinionated framework. Contributions that add structure, sharpen principles, and improve the quality of naming deliverables are welcome. Contributions that add complexity or dilute the framework's point of view are not.

## What's worth contributing

- **New naming principles** grounded in real cases -- with the case attached. Don't add a principle without evidence.
- **Improvements to skill workflows** based on what broke or was awkward in practice. Describe what happened and what the fix addresses.
- **New examples** that show the framework applied to a real (or plausibly real) naming situation.
- **Corrections to the banned vocabulary list** -- additions or removals, with the reasoning.
- **Improvements to the dimension scoring rubric** -- if a score consistently produces wrong verdicts, say why.

## What isn't a contribution

- Expanding the banned vocabulary list without a reason
- Adding caveats, disclaimers, or hedges to the verdict framework
- Making the recommendation step more neutral
- Adding more skills without a clear gap the existing skills don't cover

## How to contribute

1. Open an issue describing the change and the reason before writing code or prose.
2. Reference a real case if possible. The Array/Avra naming case is the canonical first example. New contributions are stronger when grounded in practice.
3. Keep changes minimal. A targeted edit to one skill or one principle is better than a broad refactor.
4. Submit a PR with a clear description of what changed and why.

## Formatting

Follow the voice and format conventions in the existing SKILL.md and SYSTEM.md files. Direct, specific, practitioner register. No hedging, no filler, no "it depends" without saying what it depends on.
